<SectionIntro>
Networking is where EKS gets uniquely AWS-flavored. The **VPC CNI plugin** gives every pod a real IP address from your VPC ‚Äî no overlay network. This means pods can communicate directly with other AWS services (RDS, ElastiCache, etc.) using native VPC networking and security groups.
</SectionIntro>

<SectionSubheading id="toc-vpc-cni">{'\u{1F310}'} VPC CNI</SectionSubheading>

<SectionNote>
**AWS Load Balancer Controller** is the standard way to expose EKS services. It automatically provisions ALBs (for HTTP) or NLBs (for TCP/gRPC) when you create Ingress or Service resources. Install it as an add-on.
</SectionNote>

<SectionSubheading id="toc-ingress">{'\u{1F4DD}'} Ingress example</SectionSubheading>

<CodeAccordion title="ingress.yaml ‚Äî Route traffic to your frontend" ext="yaml">
{`apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: frontend-ingress
  annotations:
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
spec:
  ingressClassName: alb
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port:
                  number: 80`}
</CodeAccordion>

<Explainer heading="Service Mesh">
For complex microservice architectures, AWS App Mesh or Istio add mTLS, traffic shaping, and observability between services. Overkill for small clusters, essential for large ones.
</Explainer>
