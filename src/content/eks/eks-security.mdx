<SectionIntro>
EKS security is built on a layered model that ties Kubernetes RBAC to AWS IAM. The most important concept to understand is **IAM Roles for Service Accounts (IRSA)** ‚Äî it lets pods assume specific IAM roles without storing credentials.
</SectionIntro>

<SectionSubheading id="toc-overview">{'\u{1F512}'} Security model</SectionSubheading>

EKS security works in layers. Each layer adds defense-in-depth so a breach at one level doesn't compromise the entire cluster.

<SectionSubheading id="toc-irsa">{'\u{1F511}'} IRSA example</SectionSubheading>

<CodeAccordion title="IRSA ‚Äî Give a pod S3 access without credentials" ext="yaml">
{`# 1. Create an IAM role with a trust policy for the K8s service account
# 2. Annotate the service account:
apiVersion: v1
kind: ServiceAccount
metadata:
  name: frontend-sa
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/FrontendS3ReadRole
# 3. Reference it in your deployment:
spec:
  template:
    spec:
      serviceAccountName: frontend-sa
      containers:
        - name: frontend
          # This pod can now call S3 APIs ‚Äî no keys needed!`}
</CodeAccordion>

<Gotcha>
**Common mistake:** Don't use the node's IAM role for pod permissions. Every pod on that node gets the same access. IRSA gives each pod its own scoped role ‚Äî principle of least privilege.
</Gotcha>
